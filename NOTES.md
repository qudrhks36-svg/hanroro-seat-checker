# 한로로 콘서트 좌석 알리미 (hanroro-seat-checker)

롯데콘서트홀 "SERIES.L : 한로로" 공연(2026-08-06, 2026-08-07) 좌석이 취소로 풀리는 순간 텔레그램으로 알려주는 자동화. 다른 PC에서 이 저장소를 열었을 때 왜 이렇게 구성되어 있는지 바로 파악할 수 있도록 정리.

## 구조

- `scripts/check_seats.py` — 롯데콘서트홀 스케줄 API(`/product/ko/performance/{id}/schedule`)를 직접 호출해서 회차별 매진 여부 확인. Playwright/브라우저 불필요. 같은 API 응답의 `Seats` 배열에 등급별(R석/S석/A석/시야방해R/S/A) 잔여석수(`BookableCount`)가 이미 포함돼 있어서, 별도 API 없이 등급별 취소표를 구분할 수 있음(2026-07-21 확인).
- `.github/workflows/check-seats.yml` — GitHub Actions 워크플로. **트리거는 `workflow_dispatch`만 있음.**
- `state.json` — 마지막으로 "좌석 없음" 알림을 보낸 시각을 기록. 워크플로가 매 실행마다 커밋해서 저장소에 영속화됨 (`chore: update seat check state [skip ci]` 커밋들).

## 스케줄링: 왜 GitHub 자체 cron이 없는가

**실제 스케줄러는 저장소 밖의 cron-job.org.** 2분 간격으로 GitHub Actions API(`workflow_dispatch`)를 호출해서 실행시킴.

- GitHub 자체 `schedule:` cron을 같이 켰다가, cron-job.org와 중복 실행되는 문제가 있어서 제거함 (2026-07-16).
- 과거에 cron-job.org 설정을 끄면서 GitHub 쪽 대체 스케줄을 안 넣어 **완전히 자동 실행이 멈췄던 사고가 있었음** — cron-job.org 대시보드에서 이 job이 살아있는지, 시간대가 KST로 맞는지 주기적으로 확인 필요.
- **둘 다 켜지 말 것.** 하나만 켜져 있어야 함 (지금은 cron-job.org 단독).

## 알림 로직 (scripts/check_seats.py)

- **9시~18시(KST) 안에서만 동작.** 이 시간 밖이면 조회조차 안 하고 조용히 종료 — cron-job.org가 오작동해서 시간대 밖에 호출해도 안전하도록 스크립트 자체에 방어 로직을 넣음.
- **예외: 취소표 반영 집중구간(burst window, 2026-07-20 추가).** 롯데콘서트홀 정책상 (1) 무통장입금 미결제 취소표는 **공연일 00:00 이후 순차 반영**(보통 00:05~00:10), (2) **인터넷 취소 마감은 공연일 전 마지막 평일 18:00**이라 그 직후 취소표가 추가로 풀릴 수 있음. 이 두 시각은 원래 9-18시 가드에 걸려 아예 조회가 안 되던 사각지대였음 — `SHOW_DATES`(공연일 목록)를 기준으로 각 공연일 `00:00~00:20`, 그 전 마지막 평일(주말은 건너뜀) `18:00~18:20` 구간에서는 `in_burst_window()`가 가드를 예외적으로 풀어준다. cron-job.org는 원래 하루 종일 2분 간격으로 호출하고 있어서, 별도로 스케줄을 더 촘촘하게 만들 필요 없이 이 가드만 풀면 그 구간에도 자동으로 2분 간격 체크가 살아남.
- **좌석 없음**: `state.json`의 마지막 알림 시각 기준 **30분 경과했을 때만** 알림 (매 2분마다 스팸 보내지 않기 위함). burst window에서도 이 스로틀은 그대로 적용됨(좌석 없음 알림 스팸 방지 목적이라 유지).
- **좌석 발생**: 스로틀 무시하고 **즉시** 알림 + 회차별 좌석 선택 URL + 등급별 잔여석 개수(예: "R석 1석, A석 2석") 포함.
- **R/S석 강조(2026-07-21 추가):** 사용자가 이미 A석을 예매한 뒤, R석/S석 취소표를 특히 노리고 싶다고 요청. `RS_CLASSES = {"R석", "S석"}`(시야방해R/S는 할인가·시야제한이라 제외) 중 하나라도 `BookableCount > 0`이면 알림 헤더가 평소 "🚨 좌석 발생!" 대신 "🌟🚨 R/S석 취소표 발생!! 🚨🌟"로 바뀜. A석만 열렸을 때는 기존과 동일하게 평범한 헤더로 계속 알림(사용자가 A석 알림도 계속 받길 원함 — 본인 외 지인 좌석용).
- **2026-08-08 00:00 KST 이후**(마지막 공연 다음날): 자동으로 종료 알림을 보내고 `disable_workflow()`로 워크플로 자체를 비활성화함 (영구 방치 방지, 비용 0원 보장).

## 결제 속도 관련 (2026-07-16 대화에서 정리됨)

- **네이버 로그인 미지원.** 롯데콘서트홀은 L.POINT 통합회원 아이디/비번으로만 로그인 가능. 네이버 계정으로 우회 불가.
- 알림에 오는 좌석 URL(`SeatPreviewProcess.aspx`)은 **로그인 없이도 좌석 배치도까지는 바로 보임.** 로그인이 필요한 건 실제 좌석 선택 후 결제 단계.
- 결제를 빠르게 하려면 **알림 URL을 열 브라우저에 미리 로그인해두고 세션을 유지**하는 게 핵심. 네이버 앱 인앱브라우저 / 텔레그램 인앱브라우저 / 시스템 기본 브라우저는 쿠키가 서로 공유되지 않으므로, 텔레그램 링크를 길게 눌러 "브라우저에서 열기"로 **로그인해둔 바로 그 브라우저**로 열어야 함.
- **아이폰 사파리 로그인 시 흰 화면 멈춤 버그가 공식 FAQ에 등록되어 있음.** 공식 해결법이 사파리 방문기록/쿠키 전체 삭제라서, 결제 직전에 이 버그가 터지면 로그인 세션도 같이 날아가는 딜레마가 있음. 미리 세션을 살려두면(로그인 과정을 다시 안 타면) 이 버그를 피할 가능성이 높음. 리스크가 걱정되면 사파리 대신 아이폰 Chrome 사용 고려.

## 트러블슈팅

- 텔레그램 알림이 안 온다 → `gh run list --workflow=check-seats.yml` 로 최근 실행 확인. 실행 자체가 없으면 cron-job.org 쪽 문제, 실행은 있는데 메시지가 없으면 30분 스로틀 또는 9-18시 밖일 가능성.
- 워크플로가 `disabled` 상태다 → CUTOFF(2026-08-08) 지났거나, 수동으로 비활성화된 것. `gh api repos/qudrhks36-svg/hanroro-seat-checker/actions/workflows` 로 상태 확인.

## 별도 자동화: 한로로 일정 알리미 (scripts/check_hanroro_schedule.py, 2026-07-20 추가)

좌석확인 봇과는 **완전히 별개의 자동화**. 목적도 다르고(특정 공연 좌석 실시간 감시 vs 한로로의 향후 공연/페스티벌 일정 자체를 찾아서 알림), 만료도 없음(좌석봇은 2026-08-08에 자동 종료, 이건 계속 돔).

- `.github/workflows/check-hanroro-schedule.yml` — 마찬가지로 `workflow_dispatch`만 있고, **cron-job.org에 별도 job을 하나 더 만들어서 매일 08:50 KST에 1회 호출**해야 함(money_agent 주식봇과 같은 시각). 이 job은 아직 사용자가 직접 등록해야 하는 상태 — 2분 간격 좌석체크 job과 헷갈리지 않게 URL을 `check-hanroro-schedule.yml`로 정확히 넣을 것.
- `OPENROUTER_API_KEY`를 이 저장소 시크릿으로 추가함(blog_agent/stock_agents와 같은 무료 키 재사용).
- 동작: 구글 뉴스 RSS로 "한로로 콘서트/페스티벌/티켓 예매" 등 여러 검색어로 기사를 모으고, 기사 본문 일부까지 가져와서 무료 OpenRouter 모델에게 "오늘 이후 일정 + 예매 링크/예매일" 추출을 시킨 뒤 텔레그램으로 보냄.
- **EXCLUDED_SHOWS**: 8월 6일·7일 시리즈L 공연은 check_seats.py가 이미 별도로 실시간 안내 중이라, 이 알리미에서는 중복 안내하지 않고 아예 빼도록 LLM에게 지시한다(사용자 요청, 2026-07-20). `last_date`가 지나면 자동으로 제외 목록에서도 빠짐(수동 삭제 불필요) — 이후엔 이 공연이 다시 검색 결과에 걸려도 정상적으로 언급됨(과거 일정이라 어차피 날짜 필터에도 걸러짐).
- **실제로 겪은 버그**: 처음엔 LLM이 "뉴스 발행일"(예: 6월 30일)과 "기사가 다루는 실제 공연 날짜"(8월)를 혼동해서 명백히 미래인 공연을 "지난 일정"으로 착각하고 스킵하는 문제가 있었음 — 프롬프트에 "발행일 ≠ 행사일" 구분을 명시적으로 설명하고 나서 해결됨. 이런 종류의 자동화를 또 만들 때 참고할 것.
- 상태 파일 없음(스로틀/중복방지 로직 없이 매일 그냥 새로 검색해서 보냄) — 좌석봇과 달리 "매일 아침 브리핑" 성격이라 굳이 필요 없다고 판단.
